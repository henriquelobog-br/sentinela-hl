# Documento 112.2C Rev 2 — Evidence Builder

Evolução do Claim Builder. Além de montar o `FilterAgentInput`, agora enriquece
com sinais objetivos de evidência. Continua **100% determinístico, sem LLM**.

> Renomeado: `ClaimBuilder → EvidenceBuilder`. O nome antigo segue válido por
> alias (`claim_builder.py` é shim) — nenhum import quebra.

## O que mudou

Antes: `epistemic_status = hypothesis` sempre. Consequência (registrada no Rev 1):
o filtro 2 (overclaim) ficava dormente porque nada superclamava.

Agora: o Builder distingue **observação/anúncio** de **estudo/interpretação** e
pode emitir `confirmed_fact` — o que reativa o filtro 2 do Agente 113.

## A regra (porta tripla — segurança primeiro)

`confirmed_fact` (confidence `0.75`) **só** quando as três condições valem:

1. **agência oficial** — domínio em `OFFICIAL_AGENCY_DOMAINS` (nasa.gov,
   noaa.gov, usgs.gov, esa.int…). **Journals não contam** (nature/science
   publicam estudos, não observações).
2. **sinal de observação** — termos como reported, recorded, announced,
   detected, magnitude, earthquake, record…
3. **ausência de sinal de estudo** — study, research, suggests, may, preliminary,
   projection, model… **vetam** o fato.

Qualquer falha nas três → `hypothesis` (default seguro). A assimetria é
proposital: na dúvida, subclama.

| Fonte | Conteúdo | Rótulo |
|---|---|---|
| noaa.gov | "announced … warmest on record" | `confirmed_fact` 0.75 |
| usgs.gov | "recorded a magnitude 6.2 earthquake" | `confirmed_fact` 0.75 |
| nasa.gov | "a new **study suggests** …" | `hypothesis` (veto de estudo) |
| nature.com | "researchers report …" | `hypothesis` (não é agência) |
| genérico | qualquer | `hypothesis` |

## Por que a porta tripla importa (segurança)

A missão do Sentinela (ADR-001) é **impedir overclaim**. Um componente
determinístico dizendo "é fato porque o domínio é oficial" seria autoridade
disfarçada de evidência — o erro que o sistema existe pra pegar. A porta tripla
garante que só uma **observação factual de agência oficial** vira fato, e mesmo
assim o rótulo é **provisório**: o Agente 113 audita e o humano confirma na
curadoria. O Builder não decide a verdade; ele dá um chute qualificado e
auditável.

## Auditoria dos sinais (`context.notes`)

Cada decisão registra o porquê:

```
signals: source=official(usgs.gov) evidence=observation(occurred) -> confirmed_fact | category=science_research keywords=6
signals: reliability=high evidence=study(study) -> hypothesis | category=science_research keywords=8
```

## Passo 7 — reexecutar o 112.4 (é teste de máquina)

O 112.4 usa o Evidence Builder automaticamente (via shim). Para o antes/depois
real com o feed da NASA:

```bash
uv run python -m sentinela.validation.run --collector rss --url <FEED_NASA>
```

Compare o `validation-*.html` de hoje com um anterior: itens de observação que
antes saíam `hypothesis` agora saem `confirmed_fact`, e o filtro 2 do agente
passa a ter o que auditar.

**Honestidade:** as regras deste documento foram **prescritas**, não observadas —
o 112.4 real (RSS + OpenRouter) não roda no ambiente onde a ferramenta foi
construída (sem rede/key). O antes/depois foi demonstrado em dados sintéticos. A
validação de que a regra **melhora** as decisões do agente é o passo 7, na sua
máquina. Se o agente começar a superclamar por causa disso, o ajuste é apertar a
porta tripla (mais termos de estudo, menos domínios oficiais) — nunca afrouxá-la.
