# Documento 112.4 Rev 2 — Pipeline Calibration

Laboratório, não funcionalidade. Calibra o pipeline com dados reais antes da
persistência. Este documento nasce de um **run real** do 112.4 (feed NASA,
GPT-5-mini) que revelou o que precisa ser calibrado.

## O achado que importa: 0 PASS

O primeiro run real deu `PASS 0 · ESCALATE 4 · REJECT 4 · ERROR 2`. O número
crítico é **0 PASS**: com dados reais, o pipeline não valida nada sozinho — tudo
rejeita ou vai pro humano. Isso não é ruído a suavizar; é sinal de que o branch
`pass` e o roteamento do `DecisionEngine` ainda não agregam valor autônomo.

### Hipótese (confirmar nos rationales do HTML)

A proveniência (filtro 1) está sendo aplicada a RSS de um jeito que quase sempre
falha, por um descompasso no que o Evidence Builder monta:

- `claim.statement` = **título** da notícia (headline, frequentemente exagerado)
- `source.excerpt` = **resumo** do *mesmo* item

O filtro pergunta "o resumo sustenta o título?". Manchete de notícia costuma
superclamar em relação ao próprio corpo — então a proveniência flaga/rejeita
**corretamente**. O filtro funciona; ele está revelando que *manchete de RSS não
é evidência autossuficiente*.

**Como confirmar:** abra `validation-*.html`, leia o rationale de proveniência
dos itens REJECT. Se disserem "o resumo não sustenta o título" (ou equivalente),
a hipótese está certa.

### A alavanca real de calibração (não é retry nem dashboard)

Enquanto a proveniência comparar headline contra um resumo raso, ela rejeita
quase tudo. Saídas, por ordem de preferência:

1. **Collector busca o corpo do artigo** (não só o `<description>` do feed) →
   proveniência passa a ter o texto real pra verificar. *Melhor opção.*
2. **Builder monta `statement` a partir do resumo**, não do headline.
3. Aceitar 0 pass autônomo (seguro, mas o branch `pass` morre e o humano revisa
   tudo).

Isto é decisão de calibração para a próxima iteração — não foi implementado aqui
porque depende de você confirmar a causa nos rationales.

## O que foi implementado agora (infra, testável offline)

### 1. `invalid_json` → ERROR = 0

Três camadas de robustez no `FilterEngine`:

- **parser de recuperação** — se `json.loads` falha, extrai o primeiro objeto
  JSON balanceado embutido em prosa/markdown (`_extract_json_object`). Resolve o
  caso comum: o modelo responde com texto + JSON.
- **retry** — se ainda falhar, refaz a chamada uma vez (`max_attempts=2`).
- **JSON Schema (opção mais forte)** — para modelos que suportam Structured
  Outputs, passe `RESPONSE_SCHEMA` via o passthrough `extra`:
  ```python
  FilterEngine(client, extra={"response_format": {"type": "json_schema", "json_schema": RESPONSE_SCHEMA}})
  ```
  GPT-5-mini via OpenRouter suporta — ligar isso deve zerar o `invalid_json` na
  origem. (Não testável aqui sem a API; ativar na sua máquina.)

### 2. Breakdown de latência por etapa

O report agora mede e agrega: `builder_ms`, `llm_ms` e o total, por item e no
resumo. Assim o gargalo fica explícito (spoiler: ~18 s/item é quase todo LLM; o
builder é <1 ms — determinístico).

### 3. Bloco de métricas no HTML

Painel `pipeline` no topo do relatório: itens, pass/escalate/reject/error, tempo
total + builder + llm, tokens, modelo, feed. Segue o kit (dark, indigo, denso,
tabular-nums, minúsculas).

2. Ler os rationales dos REJECT → confirmar a causa do 0 PASS.
