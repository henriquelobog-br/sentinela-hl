# Documento 112.2A — Collector Adapter

Primeira metade da etapa de coleta (Milestone 2). Componente Python isolado,
sem n8n. Objetivo: transformar fonte bruta em `RawItem`, validado isoladamente
antes de entrar no workflow.

> Escopo: só coleta. Sem classificação. Sem `epistemic_status`, sem
> `confidence_score`. Sem n8n. Sem Supabase.

## Responsabilidade única

O collector **só coleta**. A saída é `list[RawItem]` — o contrato do 111,
espelho da tabela `raw.items`. O rótulo epistêmico e a confiança são de outro
componente (o classifier, etapa seguinte). Misturar coleta e classificação aqui
seria o mesmo erro do n8n que decide ou do `OpenRouterClient` que aplica regra.

```
fonte (rss/api/arquivo/mock) → Collector → RawItem   [fim do 112.2A]
                                              ↓
                                          classifier → FilterAgentInput  [depois]
```

## Contrato de saída: `RawItem` (já existia)

Nada de contrato novo. O collector produz o `RawItem` do 111, com os campos que
uma coleta de fato tem: `source_id`, `external_id`, `url`, `title`,
`raw_payload`, `normalized_content`, `content_hash`, `published_at`,
`collected_at`. Note o que **não** está aqui: `epistemic_status` e
`confidence_score` — esses moram no `Claim`, não no `RawItem`. O modelo já sabia
dessa separação desde o 111.

## Estrutura

```
sentinela/collector/
├── base.py     # Protocol Collector + content_hash (dedup)
├── mock.py     # MockCollector — itens sintéticos, zero I/O
├── file.py     # FileCollector — lê JSON local (lista de objetos)
├── rss.py      # RssCollector — feedparser (URL ou XML já carregado)
└── tests/
    └── test_collectors.py
```

## O Protocol

```python
@runtime_checkable
class Collector(Protocol):
    kind: str
    def collect(self, source: Source) -> list[RawItem]: ...
```

Cada adapter implementa isso. Adicionar uma fonte nova (NewsAPI, scraper) é um
adapter novo que produz o mesmo `RawItem` — sem tocar em nada abaixo.

## Os três adapters

| Adapter | Fonte | Rede? |
|---|---|---|
| `MockCollector` | itens sintéticos in-memory | não |
| `FileCollector` | JSON local (lista de objetos) | não |
| `RssCollector` | feed RSS via `feedparser` | sim (ou XML local para teste) |

O `RssCollector.collect` aceita `raw_xml=` opcional: com ele, parseia um XML já
carregado (teste offline); sem ele, usa `source.url` (rede). O mesmo código
serve os dois caminhos.

## Nota de contrato: `Collector.kind` × `SourceKind`

`Collector.kind` é o rótulo do **adapter** (`"mock"`, `"file"`, `"rss"`).
`Source.kind` é o enum `SourceKind` do banco (`rss/api/scraper/newsletter/manual`,
do 101). Para fontes de produção coincidem (`rss`, `api`); `mock` e `file` são
adapters de desenvolvimento e **não** entram no enum SQL — não são tipos de fonte
reais. Por isso não há migration nova aqui.

## Como testar

```bash
uv run pytest sentinela/collector/tests/
```

Tudo offline: mock e file não tocam rede; o rss é testado com um XML local
(valida o parsing do feedparser sem depender de um feed externo).

## O que a validação prova

- Os adapters satisfazem o `Protocol Collector`.
- A saída é sempre `list[RawItem]` — nenhum item carrega rótulo ou confiança.
- `content_hash` presente em todos (dedup pronto para o `raw.items`).
- O RSS extrai título, `guid`, link e data de publicação do XML.

## O que NÃO foi testado aqui

A coleta RSS **com rede real** (buscar um feed público). O parsing do feedparser
foi validado com XML local; a chamada HTTP em si depende do ambiente e de um feed
de verdade — teste na sua máquina apontando `source.url` para um feed científico
real (ex.: um feed de journal ou de agência de pesquisa dentro do escopo).
