# Documento 112.1 — Workflow n8n (esqueleto)

Companion do `workflow.json` importável. O JSON monta os nós; este `.md` explica
cada um e serve de referência para reconstruir qualquer nó na UI se a importação
reclamar (o schema do n8n muda entre versões).

> Escopo: esqueleto mockado. Sem RSS. Sem Supabase. Sem regra de negócio no n8n.

## Fluxo

```
Manual Trigger -> Montar comando -> Execute Command -> Parse JSON -> Switch
                                                                       |-- pass -> log
                                                                       |-- escalate -> log
                                                                       |-- reject -> log
```

## Princípio (linha vermelha)

n8n = orquestrador. Python = inteligência. Toda decisão já foi tomada pelo
`DecisionEngine` dentro do agente 113. O único roteamento permitido é o `Switch`
lendo `report.decision`. Proibido qualquer `IF` sobre `confidence_score`,
`reliability`, `epistemic_status`. O n8n **não lê arquivo, não parseia fixture e
não conhece o conteúdo do JSON** — só executa um comando e roteia o resultado.

## Os nós

### 1. Manual Trigger
Dispara o workflow no teste. Vira `Schedule Trigger` (cron) depois — nada abaixo muda.

### 2. Montar comando (Code)
Não carrega dado nem lê arquivo. Só monta a string do comando:

```js
const which = "pass";              // "pass" | "escalate" | "reject"
const repo = $env.SENTINELA_REPO;  // portátil: definido no ambiente do n8n
const command = `cd ${repo} && cat sentinela/filters/tests/fixtures/caso_${which}.json | uv run python -m sentinela.cli.run_filters -`;
return [{ json: { which, command } }];
```

Por que assim: quem lê o fixture é o **shell** (`cat`), não o n8n. O n8n nunca
sabe onde o JSON mora nem qual o formato dele. Isso mantém a fronteira limpa **e**
prepara o 112.2 — trocar `cat fixture.json` por `python -m sentinela.collector.rss`
não muda uma linha do workflow.

### 3. Execute Command — a caixa-preta
Parâmetro `command` = `={{ $json.command }}`. O n8n executa e captura o stdout.

Onde roda: no ambiente do n8n. O projeto (`sentinela/`) e o `uv` precisam estar
disponíveis ali, e `SENTINELA_REPO` apontando para a raiz do repo. Para o
esqueleto, o mais simples é rodar o n8n na raiz do repo (`npx n8n`). n8n em
container é tema de deploy, não do 112.1.

### 4. Parse JSON (Code)
O `Execute Command` devolve `stdout` como string:

```js
const stdout = $json.stdout || "";
return [{ json: JSON.parse(stdout) }];
```

Agora `report.decision` fica acessível.

### 5. Switch (report.decision)
Três regras sobre `={{ $json.report.decision }}`, operador `string · equals`:

| Regra | Valor | Saída |
|---|---|---|
| 1 | `pass` | output 0 -> pass -> log |
| 2 | `escalate` | output 1 -> escalate -> log |
| 3 | `reject` | output 2 -> reject -> log |

Nenhuma outra condição.

### 6. Três nós de log (NoOp)
Marcam o caminho tomado. Persistência é o 112.5.

## Contrato do Execute Command (CONGELADO)

Contrato oficial entre **qualquer** orquestrador (n8n hoje, outro amanhã) e o
Agente 113. Não muda mais:

| | |
|---|---|
| Entrada | JSON (`FilterAgentInput`) via **stdin** |
| Saída | JSON (`FilterAgentOutput`) via **stdout** |
| Erro | stdout estruturado (`{ ok:false, error:{...} }`) + **exit code** (1) |

O agente nunca crasha nem chuta veredito — sempre um envelope estruturado.

Consequência: a fonte do dado é intercambiável sem tocar o agente nem o workflow.
Basta trocar o que alimenta o stdin:

```bash
cat fixture.json                         | uv run python -m sentinela.cli.run_filters -   # 112.1
uv run python -m sentinela.collector.rss | uv run python -m sentinela.cli.run_filters -   # 112.2
```

## Base64 — quando usar (não é o caso aqui)

No 112.1 o fixture está em disco, então o `cat` lê o arquivo e não há JSON dentro
do comando para escapar. **Base64 fica reservado para quando o dado vier da
memória do n8n** — aí o JSON viajaria no comando via `echo`, e base64 elimina
problemas de aspas, unicode e quebras de linha:

```bash
echo <base64_do_json> | base64 -d | uv run python -m sentinela.cli.run_filters -
```

## Como executar localmente

```bash
cd sentinela-hl
export SENTINELA_REPO="$(pwd)"
npx n8n
```

Importe `workflow.json` (Workflows -> Import from File), escolha `which` no node
`Montar comando`, execute (Test workflow) e observe o branch acionado.

## Como testar os três casos

| `which` | Branch esperado |
|---|---|
| `pass` | pass -> log |
| `escalate` | escalate -> log |
| `reject` | reject -> log |

## Como ver o stdout do Execute Command

Clique no node `Execute Command` -> aba **Output** -> campo `stdout`. É o
`FilterAgentOutput` cru:

```json
{ "ok": true, "report": { "decision": "escalate", "requires_human_review": true, "classifications": [ ... ] }, "run_id": "..." }
```

## Como validar que os três branches funcionam

Cada `which` ativa só um branch; os outros dois ficam vazios. Três execuções,
três branches — esqueleto validado.

## Ajustes prováveis na importação (honestidade)

Não é possível testar a importação na sua versão do n8n aqui. Os nós com maior
chance de precisar de retoque são o **Switch** e os **Code** (schema muda entre
versões):

- **Switch** — modo Rules, três regras sobre `{{ $json.report.decision }}`
  (equals `pass`/`escalate`/`reject`), saídas 0/1/2 nos três NoOp (seção 5).
- **Code** — o `typeVersion` pode divergir; o que importa é o corpo em JS
  (seções 2 e 4).

O `workflow.json` foi validado quanto à integridade estrutural e o pipe
`cat fixture | run_filters -` foi testado ponta a ponta. O import no n8n em si é
o que não pôde ser testado — por isso este `.md` existe.
