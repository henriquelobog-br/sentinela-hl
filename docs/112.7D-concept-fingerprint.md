# Documento 112.7D — Concept Fingerprint

## Representação conceitual determinística de eventos científicos

**Status:** Draft  
**Versão:** 1.0  
**Epic:** 113 — Scientific Intelligence  

**Dependências:**

- Documento 101 — Arquitetura do Sentinela HL
- ADR-003 — Uma chamada de LLM por item
- ADR-004 — Contratos primeiro
- ADR-007 — Scientific Prioritization Architecture
- Documento 112.5 — Persistência Supabase
- Documento 112.7A — Scientific Taxonomy
- Documento 112.7B — Research Profile

---

# 1. Objetivo

O Concept Fingerprint transforma um evento científico validado em uma representação conceitual canônica, determinística, explicável e reutilizável.

Ele responde exclusivamente à pergunta:

> **Quais conceitos científicos da taxonomia estão explicitamente representados neste evento, por quais sinais e com qual força?**

A saída do componente será consumida posteriormente por:

- Event Radar;
- Interest Engine;
- Prioritized Bulletin;
- Research Questions;
- Evidence Linking;
- Scientific Memory;
- futuro Knowledge Graph.

O Concept Fingerprint não calcula interesse pessoal e não decide importância científica.

---

# 2. Motivação

Os eventos armazenados em `knowledge.events` contêm campos estruturados, como:

- título;
- resumo;
- categoria;
- área científica;
- palavras-chave;
- entidades;
- evidências;
- localização;
- status epistêmico;
- confiança.

Esses campos ainda não constituem uma representação conceitual uniforme.

Dois eventos relacionados podem usar palavras diferentes:

```text
Evento A:
"Desert dust crossed the Atlantic"

Evento B:
"Aeolian aerosol plume reached South America"
```

A Scientific Taxonomy pode resolver:

```text
desert dust
aeolian dust
mineral aerosol
```

para conceitos canônicos como:

```text
mineral_dust
aerosols
atmospheric_transport
```

O Concept Fingerprint centraliza essa resolução para impedir que Event Radar, Interest Engine e outros componentes implementem mecanismos próprios e potencialmente divergentes.

---

# 3. Definição

Um Concept Fingerprint é uma coleção ordenada de conceitos canônicos encontrados em um evento, acompanhados por:

- peso;
- origem do sinal;
- termo encontrado;
- campo de origem;
- tipo de correspondência;
- caminho taxonômico;
- justificativa determinística.

Exemplo simplificado:

```python
ConceptFingerprint(
    event_id=event.id,
    taxonomy_version="1",
    concepts=[
        ConceptSignal(
            concept_id="mineral_dust",
            weight=1.0,
            sources=[
                MatchEvidence(
                    field="title",
                    matched_term="mineral dust",
                    match_type="canonical_name",
                    contribution=1.0,
                )
            ],
        ),
        ConceptSignal(
            concept_id="atmospheric_transport",
            weight=0.55,
            sources=[
                MatchEvidence(
                    field="keywords",
                    matched_term="trade winds",
                    match_type="related_concept",
                    contribution=0.55,
                )
            ],
        ),
    ],
)
```

# 4. Posição na arquitetura

```text
Knowledge Event validado
        │
        ▼
Text and Structured Field Normalization
        │
        ▼
Scientific Taxonomy Resolution
        │
        ▼
Concept Fingerprint
        │
        ├── Event Radar
        ├── Interest Engine
        ├── Prioritized Bulletin
        ├── Research Questions
        └── Scientific Memory
```

O componente opera depois da validação científica e antes da priorização.

---

# 5. Responsabilidade única

O Concept Fingerprint deve:

- receber um evento validado;
- examinar somente campos autorizados;
- identificar termos resolvíveis pela Scientific Taxonomy;
- converter termos em `concept_id` canônicos;
- aplicar pesos determinísticos;
- preservar evidências de cada correspondência;
- expandir relações taxonômicas apenas segundo regras explícitas;
- produzir um objeto Pydantic imutável.

O Concept Fingerprint não deve:

- consultar o Research Profile;
- calcular relevância;
- calcular significância;
- classificar eventos como `PASS`, `ESCALATE` ou `REJECT`;
- criar conceitos;
- alterar a Taxonomia;
- alterar o evento;
- consultar banco;
- chamar LLM;
- gerar embeddings;
- inferir relações científicas não registradas.

---

# 6. Nomenclatura

A V1 utilizará o nome:

```text
Concept Fingerprint
```

e não:

```text
Semantic Fingerprint
```

A expressão “semântico” poderia sugerir embeddings, modelos vetoriais ou inferência por LLM.

Na V1, a representação é construída exclusivamente por:

- conceitos canônicos;
- nomes;
- sinônimos;
- campos estruturados;
- hierarquia;
- relações curadas da taxonomia;
- regras determinísticas.

---

# 7. Entrada

A entrada principal é um `knowledge.events` representado pelo contrato Pydantic `Event`.

Campos utilizáveis na V1:

```text
title
summary
category
scientific_area
keywords
entities
evidence
country
occurred_at
epistemic_status
confidence_score
```

Nem todos os campos geram conceitos diretamente.

## 7.1 Campos textuais primários

```text
title
summary
```

São pesquisados por nomes e sinônimos taxonômicos.

## 7.2 Campos estruturados

```text
category
scientific_area
keywords
country
entities
```

Possuem peso superior ao texto livre quando já foram explicitamente classificados pelo pipeline.

## 7.3 Evidence

O campo `evidence` poderá contribuir somente quando contiver texto ou metadados claramente identificados.

A implementação não deve tentar interpretar estruturas arbitrárias.

## 7.4 Campos contextuais

```text
epistemic_status
confidence_score
occurred_at
```

Não geram conceitos.

São preservados como metadados para consumidores posteriores.

# 8. Saída

A saída principal será um modelo Pydantic imutável:

```python
class ConceptFingerprint(BaseModel):
    event_id: UUID | None
    taxonomy_version: str
    generated_at: datetime
    concepts: tuple[ConceptSignal, ...]
    unmatched_terms: tuple[UnmatchedTerm, ...]
    source_fields: tuple[str, ...]
    algorithm_version: str
```

## 8.1 ConceptSignal

```python
class ConceptSignal(BaseModel):
    concept_id: str
    domain_id: str
    weight: float
    direct_weight: float
    inherited_weight: float
    related_weight: float
    evidence: tuple[MatchEvidence, ...]
```

## 8.2 MatchEvidence

```python
class MatchEvidence(BaseModel):
    field: FingerprintField
    input_term: str
    normalized_term: str
    matched_term: str
    canonical_concept_id: str
    match_type: ConceptMatchType
    contribution: float
```

## 8.3 UnmatchedTerm

```python
class UnmatchedTerm(BaseModel):
    field: FingerprintField
    input_term: str
    normalized_term: str
    reason: str
```

Termos não resolvidos são registrados para:

- auditoria;
- melhoria futura da taxonomia;
- detecção de lacunas vocabulares.

Eles nunca criam conceitos automaticamente.

# 10. Normalização

A normalização textual deve reutilizar a mesma função canônica da Scientific Taxonomy, sempre que possível:

```python
normalize_term()
```

A normalização deve:

- converter para minúsculas;
- remover acentos;
- colapsar espaços;
- preservar números relevantes;
- remover pontuação apenas para comparação;
- não traduzir;
- não lematizar;
- não aplicar stemming;
- não substituir termos por inferência.

Exemplos:

```text
"CALIPSO"       → "calipso"
"MERRA-2"       → "merra-2"
"Espírito Santo" → "espirito santo"
"SO₂"           → forma normalizada prevista pela taxonomia
```

A normalização nunca deve alterar o texto original registrado em `MatchEvidence.input_term`.

---

# 11. Fontes de sinal e pesos-base

Os pesos abaixo são valores iniciais de contrato e devem ser configuráveis por uma estrutura versionada.

```python
class FingerprintWeights(BaseModel):
    keyword: float = 1.00
    scientific_area: float = 1.00
    category: float = 0.90
    entity: float = 0.90
    country: float = 0.85
    title: float = 0.85
    summary: float = 0.70
    evidence: float = 0.65
    parent_decay: float = 0.70
    related_decay: float = 0.40
```

Esses pesos não representam confiança científica.

Eles representam apenas a força do sinal conceitual no evento.

---

# 12. Correspondência direta

Uma correspondência direta ocorre quando um termo resolve para um conceito por:

- `concept_id`;
- nome canônico;
- sinônimo.

Exemplo:

```text
entrada: "CALIPSO"
resolução: satellite_instruments
tipo: synonym
```

Outro exemplo:

```text
entrada: "mineral dust"
resolução: mineral_dust
tipo: canonical_name
```

A contribuição inicial é:

```text
peso do campo × peso da correspondência
```

Na V1:

```text
concept_id      = 1.00
canonical_name  = 1.00
synonym         = 1.00
```

Sinônimo não recebe penalidade, pois foi explicitamente curado na Taxonomia.

---

# 13. Expansão hierárquica

Depois de identificar um conceito diretamente, o fingerprint pode incluir seus ancestrais.

Exemplo:

```text
saharan_dust
    parent → mineral_dust
    parent → aerosols
```

A contribuição hierárquica deve decair por salto:

```text
peso_ancestral = peso_direto × parent_decay^distância
```

Com `parent_decay = 0.70`:

```text
saharan_dust: 1.00
mineral_dust: 0.70
aerosols: 0.49
```

Regras:

- somente ancestrais registrados na Taxonomia;
- número máximo de saltos configurável;
- V1: máximo de três saltos;
- nenhum descendente é inferido a partir de um conceito pai;
- a origem deve ser registrada como `match_type="parent"`.

Encontrar `aerosols` não permite inferir automaticamente `mineral_dust`.

Encontrar `mineral_dust` pode permitir incluir `aerosols`, caso a Taxonomia declare essa hierarquia.

---

# 14. Expansão por conceitos relacionados

Relações `related` podem ser incluídas com peso reduzido.

Exemplo:

```text
mineral_dust
    related → atmospheric_transport
```

Peso:

```text
related_weight = direct_weight × related_decay
```

Com `related_decay = 0.40`:

```text
mineral_dust: 1.00
atmospheric_transport: 0.40
```

Regras:

- apenas um salto na V1;
- nenhuma expansão recursiva por `related`;
- relações precisam existir na Taxonomia;
- conceitos relacionados nunca recebem peso superior ao conceito de origem;
- toda expansão deve registrar o conceito que a originou.

---

# 15. Proibição de transitividade irrestrita

A V1 não deve executar caminhadas arbitrárias no grafo taxonômico.

É proibido:

```text
A related B
B related C
C related D
→ inferir D automaticamente
```

Isso poderia criar relações fracas e falsos positivos.

A V1 permite:

- ancestrais limitados;
- `related` com um salto;
- sem mistura recursiva de `parent` e `related`.

Qualquer evolução futura precisa de documento próprio e validação empírica.

---

# 16. Agregação de evidências

Um mesmo conceito pode ser encontrado em vários campos:

```text
title: "Mineral dust crosses Atlantic"
keywords: ["mineral dust", "aerosol"]
summary: "Dust plume..."
```

As contribuições não devem ser somadas sem limite.

A agregação padrão da V1 será:

```text
peso final = 1 - produto(1 - contribuição_i)
```

Essa fórmula:

- aumenta com múltiplas evidências;
- permanece entre 0 e 1;
- evita que repetições gerem score artificialmente ilimitado.

Exemplo:

```text
title: 0.85
keyword: 1.00

peso final:
1 - (1 - 0.85) × (1 - 1.00)
= 1.00
```

Outro exemplo:

```text
summary: 0.70
evidence: 0.65

peso final:
1 - 0.30 × 0.35
= 0.895
```

Cada contribuição permanece registrada separadamente.

---

# 17. Separação dos pesos

Cada `ConceptSignal` deve preservar:

```text
direct_weight
inherited_weight
related_weight
weight
```

O peso final não pode ocultar a origem.

Exemplo:

```python
ConceptSignal(
    concept_id="aerosols",
    direct_weight=0.70,
    inherited_weight=0.49,
    related_weight=0.00,
    weight=0.847,
)
```

O consumidor poderá distinguir:

- conceito explicitamente mencionado;
- conceito herdado por hierarquia;
- conceito sugerido por relação curada.

---

# 18. Extração de termos

## 18.1 Campos com valores discretos

Para:

```text
keywords
category
scientific_area
country
entities
```

cada valor é tratado como termo independente.

## 18.2 Título e resumo

Título e resumo exigem busca por termos taxonômicos dentro de texto maior.

A V1 deve usar um índice determinístico de termos da Taxonomia.

Requisitos:

- correspondência por fronteira de palavras;
- preferência por termos mais longos;
- evitar correspondência parcial interna;
- permitir termos compostos;
- evitar duplicação por sobreposição.

Exemplo:

```text
Texto:
"mineral dust aerosol transport"

Termos:
"mineral dust"
"dust"
```

Se ambos forem sinônimos do mesmo conceito, registrar apenas a melhor correspondência textual para aquele trecho.

## 18.3 Ordenação de termos

O índice deve ordenar termos por:

1. maior número de caracteres;
2. maior número de tokens;
3. ordem lexical.

Isso garante comportamento determinístico em sobreposições.

---

# 19. Termos ambíguos

A Scientific Taxonomy 112.7A já deve impedir que o mesmo termo resolva silenciosamente para conceitos diferentes.

Caso uma ambiguidade ainda seja encontrada:

- não escolher arbitrariamente;
- registrar em `unmatched_terms`;
- usar motivo `ambiguous_term`;
- não gerar conceito;
- permitir que a validação da Taxonomia seja corrigida posteriormente.

---

# 20. Termos não resolvidos

Termos não resolvidos não são erros de execução.

Eles são lacunas esperadas.

Exemplo:

```python
UnmatchedTerm(
    field="keyword",
    input_term="new atmospheric river index",
    normalized_term="new atmospheric river index",
    reason="not_in_taxonomy",
)
```

Porém, um excesso de termos não resolvidos pode indicar:

- taxonomia insuficiente;
- evento fora dos domínios atuais;
- baixa qualidade da estrutura do evento;
- necessidade de expansão vocabular.

O componente não decide qual dessas hipóteses é verdadeira.

---

# 21. Campos de entidades

O contrato de `entities` pode evoluir.

Na V1, o extractor deve aceitar somente formas explicitamente suportadas:

```python
["NASA", "CALIPSO", "South Atlantic"]
```

ou:

```python
[
    {"name": "CALIPSO", "type": "instrument"},
    {"name": "South Atlantic", "type": "region"},
]
```

Estruturas desconhecidas:

- não causam crash;
- são ignoradas de forma explícita;
- geram diagnóstico técnico quando apropriado.

Nenhum JSON arbitrário deve ser convertido em texto recursivamente sem controle.

---

# 22. Evidência e proveniência

Todo conceito encontrado deve ser auditável até o evento original.

`MatchEvidence` deve informar:

```text
campo de origem
termo original
termo normalizado
termo taxonômico que casou
conceito canônico
tipo da correspondência
contribuição numérica
```

Exemplo:

```python
MatchEvidence(
    field="title",
    input_term="Hubble observes mineral dust",
    normalized_term="hubble observes mineral dust",
    matched_term="mineral dust",
    canonical_concept_id="mineral_dust",
    match_type="canonical_name",
    contribution=0.85,
)
```

---

# 23. Ordenação da saída

A saída deve ser estável.

`ConceptSignal` deve ser ordenado por:

1. `weight` decrescente;
2. `direct_weight` decrescente;
3. `concept_id` crescente.

As evidências internas devem ser ordenadas por:

1. campo;
2. contribuição decrescente;
3. termo normalizado;
4. `concept_id`.

`UnmatchedTerm` deve ser ordenado por:

1. campo;
2. termo normalizado.

O mesmo evento, taxonomia e configuração devem produzir JSON idêntico.

---

# 24. Versionamento

O fingerprint depende de três versões:

```text
taxonomy_version
algorithm_version
weights_version
```

Modelo recomendado:

```python
class FingerprintVersion(BaseModel):
    taxonomy_version: str
    algorithm_version: str
    weights_version: str
```

Alterações em qualquer uma dessas dimensões podem produzir fingerprints diferentes.

A reprodutibilidade exige que todas sejam armazenadas.

---

# 25. Idempotência

Executar o fingerprint várias vezes sobre o mesmo evento deve produzir o mesmo conteúdo lógico.

O campo `generated_at` pode mudar, mas não deve participar de:

- comparação lógica;
- hash determinístico;
- deduplicação.

É recomendado produzir:

```python
fingerprint_hash: str
```

calculado sobre uma serialização canônica que exclui:

- `generated_at`;
- metadados operacionais transitórios.

O hash permite:

- detectar mudanças;
- evitar persistência redundante;
- rastrear reprocessamentos.

---

# 26. Hash canônico

O hash deverá utilizar:

```text
SHA-256
```

sobre JSON:

- codificado em UTF-8;
- chaves ordenadas;
- sem espaços irrelevantes;
- listas já ordenadas;
- sem timestamp de geração.

Exemplo conceitual:

```python
sha256(
    canonical_json(
        {
            "event_id": ...,
            "taxonomy_version": ...,
            "algorithm_version": ...,
            "concepts": ...,
            "unmatched_terms": ...,
        }
    )
)
```

---

# 27. Configuração

Os pesos e limites devem ser representados por contrato Pydantic:

```python
class FingerprintConfig(BaseModel):
    version: str = "1"
    weights: FingerprintWeights
    include_parents: bool = True
    max_parent_depth: int = 3
    include_related: bool = True
    max_related_depth: int = 1
    minimum_contribution: float = 0.05
```

A configuração não deve ficar espalhada como constantes mágicas no código.

Pode existir inicialmente em:

```text
config/concept_fingerprint.yaml
```

Exemplo:

```yaml
version: "1"

weights:
  keyword: 1.00
  scientific_area: 1.00
  category: 0.90
  entity: 0.90
  country: 0.85
  title: 0.85
  summary: 0.70
  evidence: 0.65
  parent_decay: 0.70
  related_decay: 0.40

include_parents: true
max_parent_depth: 3

include_related: true
max_related_depth: 1

minimum_contribution: 0.05
```

# 32. Relação com o Event Radar

O Event Radar receberá:

```text
Event
+
ConceptFingerprint
```

O fingerprint informa:

> Quais conceitos estão presentes?

O Radar responde:

> Qual é a significância científica objetiva do evento?

O Concept Fingerprint não define:

```text
LOW
MODERATE
HIGH
CRITICAL
```

Essa responsabilidade pertence ao Event Radar.

---

# 33. Relação com o Research Profile

O Concept Fingerprint é independente de qualquer pesquisador.

O mesmo evento gera o mesmo fingerprint para todos os perfis.

Depois:

```text
Concept Fingerprint
+
Research Profile
=
Interest Result
```

Essa separação permite:

- múltiplos pesquisadores;
- reuso;
- cache;
- auditoria;
- comparação de perfis;
- menor custo computacional.

---

# 34. Relação com o Interest Engine

O Interest Engine não deve examinar diretamente título ou resumo para repetir extração.

Ele consumirá:

- conceitos canônicos do fingerprint;
- pesos e evidências;
- Research Profile;
- resultados do Event Radar.

Assim, a lógica de resolução taxonômica permanece em um único componente.

---

# 35. Relação com o Agente 113

O Concept Fingerprint não substitui o Agente 113.

O Agente 113 responde:

> A afirmação está cientificamente bem sustentada e calibrada?

O Concept Fingerprint responde:

> Quais conceitos científicos estão representados neste evento validado?

O fingerprint só deve ser produzido para eventos que alcançaram o estado apropriado no pipeline.

Um evento `rejected` não deve ser priorizado.

Eventos `escalated` podem gerar fingerprint, mas consumidores precisam preservar `requires_human_review`.


Essa tabela não deve ser criada neste documento sem decisão explícita e migration revisada.

# 37. Performance

O índice da Taxonomia deve ser construído uma vez e reutilizado.

Evitar:

- percorrer todos os conceitos para cada palavra;
- recompilar regex para cada evento;
- consultar arquivo YAML durante cada fingerprint;
- carregar Taxonomia repetidamente.

Estratégia:

```text
startup
→ carrega Taxonomia
→ valida
→ constrói índice
→ constrói matcher textual
→ processa N eventos
```

Complexidade esperada deve ser proporcional ao tamanho do evento e ao número de correspondências, não ao produto completo entre todos os termos e conceitos.

---

# 38. Segurança e limites

Textos de entrada podem ser grandes.

A implementação deve possuir limites configuráveis:

```text
maximum_title_length
maximum_summary_length
maximum_keywords
maximum_entities
maximum_evidence_items
maximum_evidence_text_length
```

O truncamento, quando necessário:

- deve ser determinístico;
- deve ser registrado;
- não pode ocorrer silenciosamente.

# 45. Riscos

## 45.1 Taxonomia insuficiente

Muitos termos podem permanecer não resolvidos.

Mitigação:

- registrar `unmatched_terms`;
- revisar métricas;
- expandir a Taxonomia manualmente;
- nunca criar conceitos automaticamente.

## 45.2 Relações excessivas

Expansão por `related` pode introduzir ruído.

Mitigação:

- apenas um salto;
- peso reduzido;
- preservar peso direto separadamente;
- permitir desativação por configuração.

## 45.3 Falsos positivos textuais

Termos curtos podem casar dentro de palavras maiores.

Mitigação:

- fronteiras de tokens;
- preferência por termos longos;
- testes de sobreposição.

## 45.4 Pesos arbitrários

Pesos iniciais são hipóteses de engenharia.

Mitigação:

- configuração versionada;
- explicabilidade;
- benchmark;
- futura calibração com dados revisados por humano.

## 45.5 Confusão entre conceito e importância

A presença forte de um conceito não significa que o evento seja importante.

Mitigação:

- fingerprint apenas representa;
- Event Radar mede significância;
- Interest Engine mede relevância.

---

# 46. Invariantes vinculantes

1. O mesmo evento, Taxonomia e configuração produzem o mesmo fingerprint.
2. Todo conceito do fingerprint existe na Taxonomia.
3. Todo peso possui origem auditável.
4. Conceito relacionado nunca é apresentado como correspondência direta.
5. Ancestral nunca é apresentado como termo explicitamente encontrado.
6. Termos desconhecidos não criam conceitos.
7. O componente não conhece o pesquisador.
8. O componente não decide relevância.
9. O componente não decide significância.
10. O componente não usa LLM.
11. O componente não usa embeddings.
12. O componente não altera conhecimento persistido.

# 48. Resultado arquitetural

Com o 112.7D concluído, o pipeline científico passa a possuir uma representação conceitual comum:

```text
Event validado
        │
        ▼
Concept Fingerprint
        │
        ├── conceitos explícitos
        ├── conceitos ancestrais
        ├── relações curadas
        ├── pesos explicáveis
        ├── proveniência
        └── termos não resolvidos
```

Essa representação será a base única para os componentes de inteligência científica.

O Event Radar não precisará reexaminar o texto para descobrir conceitos.

O Interest Engine não precisará implementar resolução taxonômica própria.

As Research Questions poderão relacionar evidências usando IDs canônicos.

A Scientific Memory poderá comparar eventos ao longo do tempo.

O futuro Knowledge Graph poderá reutilizar relações já explicitadas sem transformar inferências frágeis em fatos.

---

# 49. Princípio final

> **O Concept Fingerprint não interpreta o significado científico profundo de um evento. Ele registra, de forma determinística e auditável, quais conceitos do vocabulário controlado estão presentes ou relacionados segundo regras previamente aprovadas.**

Essa limitação é deliberada.

Ela preserva o modo de falha seguro do Sentinela HL, mantém o componente reproduzível e cria uma fundação confiável para as camadas posteriores de significância, relevância e conexão de evidências.
