# Documento 112.2C — Claim Builder

Fecha a lacuna `RawItem → FilterAgentInput`. Componente **mecânico**: aplica
sinais determinísticos e um rótulo provisório conservador. **Sem LLM.** Quem
classifica de verdade é o Agente 113.

> Ordem da Milestone 2: 112.2A (collector) → **112.2C (builder)** → 112.2B (n8n).

## Por que Builder, não Classifier

Um classifier via LLM significaria **duas** chamadas de LLM por item (classifier
+ agente). Com 1000 itens/dia, 2000 chamadas — metade desperdiçada. O Agente 113
já é o classificador real (proveniência, rótulo, calibração). O Builder só monta
o contrato:

```
Collector → RawItem → [Claim Builder] → FilterAgentInput → Agente 113 → Decision
                          mecânico          (1 chamada de LLM, só aqui)
```

## Decisão consciente: `epistemic_status = hypothesis` sempre (V1)

O rótulo sai sempre `hypothesis`. Consequência honesta: o **filtro 2 (overclaim)
fica dormente** no pipeline real — se nada entra como `confirmed_fact`, não há
overclaim a pegar. Na prática, para dado coletado, o agente atua sobretudo na
**proveniência** (filtro 1). É a escolha segura de V1: o sistema **subclama** em
vez de superclamar (modo de falha certo em ciência). Fatos genuinamente
confirmados são promovidos pelo Henrique na curadoria, não pelo pipeline. Um
`SmartClassifier` opcional (Milestone 5/6) pode melhorar o chute inicial sem
tocar no resto.

## Sinais (`signals.py`) — todos determinísticos e auditáveis

| Sinal | Regra |
|---|---|
| `derive_reliability(url)` | `high` se domínio na allowlist (nature, nasa, noaa…) · `low` se sem URL ou domínio genérico (medium.com, wordpress…) · `medium` caso contrário |
| `confidence_for(tier)` | high `0.60` · medium `0.50` · low `0.40` |
| `derive_category(text)` | `science_research` se houver termo (study, research, paper, journal, climate, data…) · `general` caso contrário |
| `extract_keywords(text)` | palavras ≥4 chars, sem stopwords, sem repetição, até 10 |
| `statement_from(title, excerpt)` | `title` se existir · senão a primeira frase do `excerpt` |

A allowlist e as stopwords são conjuntos no topo de `signals.py` — extensíveis
sem tocar na lógica.

## Mapeamento (`ClaimBuilder.build(raw_item)`)

| FilterAgentInput | Origem |
|---|---|
| `claim.statement` | `statement_from(title, excerpt)` |
| `claim.epistemic_status` | `hypothesis` (fixo, V1) |
| `claim.confidence_score` | `confidence_for(reliability)` |
| `claim.source_reliability` | `derive_reliability(url)` |
| `claim.category` | `derive_category(title + excerpt)` |
| `claim.keywords` | `extract_keywords(title + excerpt)` |
| `claim.raw_item_id` | `raw_item.id` (ou placeholder se ainda não persistido) |
| `source.excerpt` | `normalized_content` / `summary` / `content` / `title` |
| `source.url` / `source.title` / `source.reliability` | do RawItem / sinal |
| `context.case_id` | `external_id` ou `content_hash` |
| `context.scope` | `science_research` (fixo) |
| `context.notes` | resumo dos sinais aplicados (auditoria) |

## Auditoria dos sinais

Os sinais aplicados ficam registrados em `context.notes`, ex.:

```
signals: reliability=high confidence=0.6 category=science_research keywords=7
```

Assim qualquer `FilterReport` pode ser rastreado até os sinais que montaram sua
entrada — sem precisar re-executar o Builder.
