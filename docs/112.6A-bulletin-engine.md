# Documento 112.6A — Bulletin Engine

Transforma conhecimento validado em **conhecimento organizado**. Não renderiza,
não publica, não decide o que é publicável.

> Sem HTML. Sem WordPress. Sem WhatsApp. Sem LLM. Sem escrita no banco.

## Fluxo

```
knowledge.events → PostgresEventReader → list[Event] → BulletinEngine → BulletinModel
                        (só leitura)                      (determinístico)
```

O `BulletinModel` é conhecimento estruturado — pronto para alimentar qualquer
canal depois (ADR-006).

## Arquivos

```
sentinela/bulletin/
├── models.py           # BulletinModel / BulletinSection / BulletinItem
├── postgres_reader.py  # leitura pura de knowledge.events
├── engine.py           # agrupa · ordena · dedup → BulletinModel
└── tests/
```

## Regras (todas determinísticas)

**Elegibilidade** (no reader): `pipeline_status ∈ (validated, escalated)` e
`review_decision <> rejected`.

**Ordenação:** `requires_human_review` ASC → `confidence` DESC (nulls last) →
`occurred_at` DESC (nulls last) → `created_at` DESC.

**Agrupamento:** `scientific_area` → fallback `category` → fallback `outros`.

**Deduplicação (só dentro do boletim):** `primary_claim_id` quando houver; senão
título normalizado (minúsculas, sem acento, sem pontuação) + categoria.
Duplicatas viram `source_count` no item. **Nunca apaga nem atualiza o banco.**

## O que o engine NÃO faz

- não decide o que é publicável — `requires_review` é **preservado** no
  `BulletinItem` para a curadoria decidir depois. Um evento pode estar correto
  cientificamente e ainda exigir revisão: **conhecimento validado ≠ decisão
  editorial**;
- não escreve em `knowledge.events` (consumidor, nunca produtor);
- não chama LLM, não acessa rede, não conhece HTML.
